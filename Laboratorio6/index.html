Preguntas y Respuestas

1. ¿Qué ventajas ofrecen las variables CSS sobre los valores hardcodeados?

Las variables CSS permiten centralizar valores reutilizables (colores, tamaños, espaciados) en un solo lugar. Al cambiar una variable, el cambio se aplica en todo el documento automáticamente, lo que facilita el mantenimiento, reduce errores y hace el código más legible y escalable.

2. ¿En qué casos conviene declarar variables CSS fuera del selector :root?

Cuando la variable solo aplica a un componente específico. Por ejemplo, declarar --card-bg dentro de .skill-card limita su alcance a ese elemento y sus hijos. También es útil para sobrescribir variables en contextos como modo oscuro (body.dark) o dentro de media queries.

3. ¿Cómo se pueden utilizar las variables CSS para implementar un sistema de temas (como modo oscuro)?

Se declaran las variables en :root con los valores del tema claro. Luego, en un selector alternativo como body.dark, se redeclaran las mismas variables con los valores del tema oscuro. Al agregar o quitar esa clase con JavaScript, el navegador actualiza todos los elementos que usan esas variables automáticamente.

4. ¿Qué sucede cuando intentas usar una variable CSS que no ha sido declarada?

El navegador usa el valor de respaldo si se proporcionó uno: var(--color-inexistente, red). Si no hay valor de respaldo, la propiedad se comporta como invalid y se usa el valor inicial o heredado de esa propiedad, sin mostrar ningún error visible en pantalla.

5. ¿Cuál es la diferencia entre los enfoques mobile-first y desktop-first?

Mobile-first escribe los estilos base para pantallas pequeñas y usa min-width en los media queries para agregar estilos conforme crece la pantalla. Desktop-first hace lo contrario: estilos base para pantallas grandes y usa max-width para ajustar en pantallas pequeñas. Mobile-first es el enfoque recomendado hoy porque prioriza el rendimiento en dispositivos con menos recursos.

6. ¿Qué breakpoints son los más comunes y por qué?

Los más usados son 480px (móviles grandes), 768px (tablets), 1024px (laptops) y 1280px (escritorios). Se eligieron porque coinciden con los tamaños de pantalla más frecuentes en el mercado. Frameworks como Tailwind y Bootstrap usan variantes de estos mismos valores como referencia estándar.

7. ¿Cómo afecta el viewport meta tag al diseño responsivo?

Sin el meta viewport, los navegadores móviles simulan una pantalla de escritorio (generalmente 980px) e ignoran los media queries. Con <meta name="viewport" content="width=device-width, initial-scale=1.0"> se indica al navegador que use el ancho real del dispositivo, lo que hace que los media queries y el diseño responsivo funcionen correctamente.

8. ¿Qué técnicas además de media queries existen para crear diseños responsivos?

Flexbox y CSS Grid adaptan el layout de forma fluida sin necesidad de breakpoints. Las unidades relativas como %, em, rem, vw y vh escalan con el contenedor o la pantalla. La función clamp() ajusta valores entre un mínimo y máximo de forma continua. También existen los container queries, que responden al tamaño del contenedor padre en lugar del viewport.

9. ¿Cómo puedes identificar qué regla CSS está siendo aplicada cuando hay múltiples selectores?

Con las DevTools del navegador (F12), en la pestaña Styles se muestran todas las reglas que aplican a un elemento ordenadas por especificidad. Las reglas sobreescritas aparecen tachadas. El navegador aplica la de mayor especificidad; en caso de empate, la que aparece más abajo en el código.

10. ¿Qué información proporciona la pestaña Computed en DevTools?

Muestra el valor final calculado de cada propiedad CSS aplicada a un elemento, después de resolver herencias, variables, unidades relativas y cascada. Es útil para ver el valor real en píxeles de propiedades como padding, margin o font-size, aunque hayan sido definidas con rem o porcentajes.

11. ¿Cómo puedes simular diferentes dispositivos sin tener acceso físico a ellos?

Con el modo de dispositivo en las DevTools del navegador (ícono de móvil en Chrome/Firefox). Permite seleccionar dispositivos predefinidos, cambiar el tamaño del viewport manualmente, simular densidad de píxeles y emular conexiones lentas. También existen herramientas como Responsively App o BrowserStack.

12. ¿En qué situaciones es preferible usar Bootstrap sobre Tailwind CSS y viceversa?

Bootstrap es preferible cuando se necesita prototipar rápido con componentes prediseñados listos (modales, navbars, tablas) o cuando el equipo tiene poca experiencia en CSS. Tailwind es mejor cuando se quiere control total sobre el diseño sin sobrescribir estilos, se prioriza un bundle pequeño en producción o se trabaja en proyectos con diseño muy personalizado.

13. ¿Qué significa que Tailwind sea "utility-first"?

Significa que el diseño se construye aplicando directamente clases de utilidad pequeñas y de propósito único en el HTML, como text-center, bg-blue-500 o p-4, en lugar de escribir clases semánticas con estilos en un archivo CSS separado. Cada clase hace una sola cosa y se combinan para lograr cualquier diseño.

14. ¿Cómo afecta el uso de un framework CSS al tamaño final del bundle?

Bootstrap incluye todos sus estilos por defecto, sumando 150-200kb sin minificar. Tailwind en desarrollo es enorme, pero en producción usa un proceso de purge que elimina todas las clases no usadas, dejando archivos de pocos kilobytes. Un framework bien configurado puede ser más ligero que CSS escrito a mano sin optimizar.

15. ¿Es posible combinar múltiples frameworks CSS en un mismo proyecto? ¿Por qué sí o por qué no?

Técnicamente es posible, pero no se recomienda. Combinar frameworks aumenta el peso del bundle, puede generar conflictos de especificidad entre sus estilos y dificulta el mantenimiento al mezclar convenciones distintas. Solo tiene sentido de forma temporal durante una migración gradual de un framework a otro.
